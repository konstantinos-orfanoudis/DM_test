$Logger.info() vs $Logger.Info() Style
NLog method call casing — widespread across all exporter & parser files

PowerShell dispatches .NET method calls case-insensitively, so both .info() and .Info() work at runtime. However, the codebase mixes both forms with no consistent convention. All 8 *_Main_PsModule.psm1 files were standardised to .Info() during the reformat, but the exporter and parser files beneath them still use lowercase.

Scripts_Exporter_PsModule.psm1 TableScripts_Exporter_PsModule.psm1 FormatScripts_Exporter_PsModule.psm1 CanSeeScripts_Exporter_PsModule.psm1 CanEditScripts_Exporter_PsModule.psm1 DBObjects_XmlParser.psm1 DBObjects_XmlExporter.psm1 DBObjects_CsvExporter.psm1 DBObjects_FilterColumnsPsModule.psm1 Process/DmDoc.psm1 Process/Export-Process.psm1 
# Current (mixed — both work, but inconsistent)
$Logger.info("The UID_DialogColumn found is: $s")
$Logger.Info("Export completed successfully!")

# Recommended — match the NLog API surface and the Main modules
$Logger.Info("The UID_DialogColumn found is: $s")
Fix: Global search-and-replace $Logger.info( → $Logger.Info( across all .psm1 files. No logic change — purely cosmetic.

PowerShell cmdlet casing — 2 exporter files

The custom Get-Logger function is called as Get-logger (lowercase l) in several places. PowerShell cmdlets are case-insensitive, so this works. However, it creates visual inconsistency and makes grep-based searches unreliable.

DBObjects_CsvExporter.psm1 DBObjects_FilterColumnsPsModule.psm1

$Logger.info("CSV ExcludedColumns file found. Importing...")

# Recommended
$Logger = Get-Logger
Fix: Search-and-replace Get-logger → Get-Logger in the two affected files.
3
$session assigned but never used (6 of 8 Main modules) Linter
PSUseDeclaredVarsMoreThanAssignments — Scripts, TableScripts, FormatScripts, CanSeeScripts, CanEditScripts, Templates

Connect-OimPSModule returns a session handle. In DBObjects and Process this handle is genuinely consumed — passed to Get-ForeignKeyMetadataPsModule, Get-ColumnPermissionsPsModule, and GetAllProcessFromChangeLabel.

In the other 6 modules the call is made purely for its side effect: loading the DLL and establishing credentials so that internal SQL helpers can use the connection. The returned $session is never referenced again, which triggers the PSUseDeclaredVarsMoreThanAssignments PSScriptAnalyzer warning and misleads readers into thinking the session object is used downstream.

Scripts_Main_PsModule.psm1 TableScripts_Main_PsModule.psm1 FormatScripts_Main_PsModule.psm1 CanSeeScripts_Main_PsModule.psm1 CanEditScripts_Main_PsModule.psm1 Templates_Main_PsModule.psm1 
# Current — $session stored but never passed anywhere
$session = Connect-OimPSModule -DMConfigDir $DMConfigDir -DMDll $DMDll ...
$Logger = Get-Logger
Write-ScriptsAsVbNetFiles -Scripts $scripts -OutDir $outDirScripts  # no $session here

# Recommended — assign to $null to document the intentional discard
$null = Connect-OimPSModule -DMConfigDir $DMConfigDir -DMDll $DMDll ...
$Logger = Get-Logger
Write-ScriptsAsVbNetFiles -Scripts $scripts -OutDir $outDirScripts
Fix: Replace $session = Connect-OimPSModule with $null = Connect-OimPSModule in the 6 affected modules. This silences the linter warning and explicitly signals to readers that the return value is intentionally discarded.

Mixed counter increment style across 7 exporter files

The codebase uses two different idioms to increment a counter variable. Most exporter files use the verbose assignment form; Process_Main_PsModule.psm1 and DBObjects_CsvExporter.psm1 use the compact ++ operator.

Join-Path -Path X -ChildPath Y vs Join-Path X Y Style
Named-parameter vs positional form — 5 Main modules vs 2

Five of the Main modules use explicit -Path / -ChildPath parameter names; the two more recently written modules (DBObjects and Process) use the shorter positional form. Both are functionally identical. 

# Current — verbose named parameters (Templates, FormatScripts, TableScripts, CanSeeScripts, CanEditScripts, Scripts)
$outDirScripts = Join-Path -Path $OutPath -ChildPath "Scripts"

# Current — short positional form (DBObjects, Process)
$outpathfolder = Join-Path $OutPath "DBObjects"

# Recommended — standardise on the short form
$outDirScripts = Join-Path $OutPath "Scripts"
Fix: Normalise all Join-Path -Path X -ChildPath Y calls to Join-Path X Y in the 5 affected Main modules.
6
Redundant $Logger = Get-Logger calls inside catch / if blocks Redundant
42 total Get-Logger calls across 8 Main modules; many are unnecessary

Get-Logger must be called after Connect-OimPSModule because the DLL load reinitialises NLog and the previous logger handle becomes stale. That one re-call is intentional and must be kept.

All other repeated calls within the same try/catch scope — inside if blocks that do not call Connect-OimPSModule, inside catch blocks, inside nested if blocks in catch — are redundant. The $Logger from the outer scope is still valid there. 

# Pattern found in every Main module's catch block
catch {
  $Logger = Get-Logger        # ← redundant — logger not invalidated by Connect-OimPSModule in catch scope
  $Logger.Info("ERROR: Export failed!")
  ...
  if ($_.ScriptStackTrace) {
    $Logger = Get-Logger      # ← doubly redundant — same catch scope, no DLL reload
    $Logger.Info("Stack Trace:")
  }
}

# What's actually needed
catch {
  # $Logger from try scope is still valid — no re-fetch needed
  $Logger.Info("ERROR: Export failed!")
  ...
  if ($_.ScriptStackTrace) {
    $Logger.Info("Stack Trace:")
  }
}

# The ONE re-call that IS required (after DLL load reinitialises NLog)
$session = Connect-OimPSModule ...
$Logger = Get-Logger    # ← required — DLL load reset NLog
Fix: Remove all $Logger = Get-Logger calls that are NOT immediately following Connect-OimPSModule. This reduces noise in the call stack and makes the one mandatory re-fetch stand out clearly.
